# CHANGELOG

## 1.3.23

- Homebrew ZIP export now saves directly into the world folder (`Data/worlds/<world>/homebrew-exports/`) via Foundry's file API instead of triggering a browser blob download — the desktop/Electron app was prompting to "open with an app" rather than saving the blob link. A toast reports the saved path; if the upload fails (e.g. no file-upload permission) it falls back to the browser download. Uses the v13+ namespaced `FilePicker` with a v12 global fallback

## 1.3.22

- **Homebrew ZIP export now works on Foundry v12**: the "Export Folders as ZIP" feature relied on a global `JSZip`, which only exists in v13+ (v12 reported "ZIP export is unavailable — JSZip not found"). Replaced it with a self-contained ZIP writer (STORE method + CRC-32) so the export works identically on v12/v13/v14 with no external dependency. Output verified to pass archive integrity checks
- Release workflow fixes: removed the redundant `generateReleaseNotes: true` from the release action (it was throwing `403: Resource not accessible by integration` against the generate-notes API, while a full `bodyFile` was already supplied); deleted the legacy `releaseOnPullRequest.yml` workflow that ran on every push and raced the advanced workflow for the same tag; bumped `actions/checkout` and `actions/setup-node` from `@v4` to `@v5` to clear the Node.js 20 deprecation warning

## 1.3.21

- **Dedicated "Homebrew" compendium**: registered a new `homebrew` Item pack (`packs/homebrew`, built from `src/packs/homebrew/`) so community/homebrew content lives in its own compendium, separate from the official packs. The release workflow excludes the raw `submissions/` folder from the distributed zip
- **Homebrew Laboratory — "Export Folders as ZIP"**: new control on the Import/Export tab. Pick the Item folders holding your homebrew and download a ZIP. Linked children travel with their parents (a blasphemy's `blasphemyPower`s, an agenda's tasks/abilities, a sin mark's abilities) even when filed in different folders, fixing the broken cross-references the old single-file export produced
  - The ZIP is laid out like the system's own pack sources — every item becomes a `src/packs/homebrew/<Name>_<id>.json` file with a `!items!<id>` key (matching `tools/unpack.mjs`), plus a root `_manifest.json`. It drops straight into the repo and compiles via `npm run build:packs`
- **Homebrew submission pipeline**: added a `submissions/` drop-folder with contributor instructions, a `tools/validate-submissions.mjs` validator (checks JSON validity, known item types, and required fields), and a `validate-submissions.yml` GitHub Action that runs the validator on any PR touching `submissions/**`

- CAT level strip: extended from 7 slots (CAT 0–VI) to 8 slots (CAT 0–VII) and fixed the missions-survived mapping. Now reads `0, 0, 1, 2, 4, 7, X, X` under CAT 0 through VII respectively, matching the CAIN rulebook
- Release workflow: the "Notify Foundry VTT Package Release API" step now reads `compatibility.minimum/verified/maximum` from `system.json` via `jq -r` instead of hardcoding `"13"`. Bumping the verified field in `system.json` will now propagate to the foundryvtt.com listing on the next release
- **Per-character blasphemy powers** (enables reflavoring without affecting other characters):
  - When a player adds a blasphemy power (via the Add Power dropdown, dragging from the sidebar, or by adding a parent Blasphemy that auto-adds passives), the system now clones the catalog item onto the actor as an embedded copy and stores the embedded id in `currentBlasphemyPowers`. Each character has their own copy that can be edited from the character sheet's Items tab without touching the world-level catalog or other characters' versions
  - Clones are tagged with `flags.cain.originWorldId` so the system can group them under the right parent Blasphemy and prevent duplicate adds
  - Removing a power (or removing its parent Blasphemy) now also deletes the embedded copy to avoid leaking orphan items
  - **Migration: none required.** Existing characters keep working — `_getItemsFromIDs` falls back from `actor.items.get(id)` to `game.items.get(id)`, so legacy world-id references still resolve and render identically. Only powers added *after* the update become per-character editable
  - Added a GM-only header button (📋 icon) on the character sheet: **"Convert blasphemy powers to per-character copies"**. Click to migrate that character's existing powers to embedded copies on demand. Idempotent and per-actor — bad runs only affect one character. Dead references (catalog items that have since been deleted) are silently dropped from the array
  - `_sendBlasphemyPowerMessage` (the print-to-chat arrow next to each power) was looking up the item only via `game.items.get(id)`, which returned `undefined` for newly embedded copies and crashed on `.system`. Same `actor.items.get(id) ?? game.items.get(id)` fallback applied; a missing power now shows a warning toast instead of a TypeError
  - Bond high-blasphemy auto-grant ([`_updateBondLevel`](module/sheets/actor-sheet.mjs)): crossing a bond's threshold up now embeds the high-blasphemy power on the actor instead of pushing the world id; crossing back down removes by origin match and deletes the embedded doc
  - Homebrew "search and add Blasphemy" path: same embed-on-add for the parent blasphemy's auto-added passive powers as the drag-drop path. Keeps both add routes producing the same data shape
- GM Convert button: swapped icon to `fa-user-pen` (more semantic for per-user editing) and set both `title` and `data-tooltip` attributes so Foundry's native styled tooltip shows on hover
- Hunt Tracker ↔ Global Talisman: changes made on the Hunt Tracker now push back to the linked Global Talismans (the inverse direction was missing). Wired via the `currentHunt` setting's `onChange`. Both sync functions are no-op when the values already match, so the bidirectional sync can't feedback-loop
- Pressure → Execution scaling: when pressure increases (via the +1 button, "tension filled" cascade, or any rule that bumps pressure), the linked execution talisman's `maxMarkAmount` is now pushed to `8 + pressure + CAT` so its visible size grows in lockstep. Decrementing pressure shrinks the talisman; if `currMarkAmount` would exceed the new max it's clamped down and the image path is updated to match
- Talisman tile size: added two world-scope settings — **Talisman Tile Width (px)** and **Talisman Tile Height (px)** — defaulting to 274×432 (the prior hardcoded values). Larger scenes can bump these so the Create Tile button produces an appropriately-sized tile instead of needing a manual resize each time
- Talisman completion visual: when `currMarkAmount >= maxMarkAmount`, the marks readout in the Global Talismans window now turns CAIN-orange with a glow + pulsing "✓ FULL" badge, and the slider accent matches. Makes "this talisman is done" obvious at a glance instead of requiring you to read the fraction
- Talisman tile label alignment: the "Create Tile" label was a fixed 200px wide regardless of the (now configurable) tile width, so it sat slightly off under the talisman. Label width now matches the tile width, font size scales mildly with the tile, and the left edge anchors to the tile's left edge instead of relying on horizontal centering math. Looks flush whether you keep the 274 default or bump to 800+
- Talisman tile label move-handler: the `updateTile` hook recomputed the label position whenever the tile was moved/resized — and used the *old* centering math with `label.shape?.width || 200`, which would read stale or undefined values and shift the label sideways and downwards from where the creation code put it. Now centers the label horizontally directly below the tile using the LIVE tile width, and resyncs the label's `shape.width` / `shape.height` / `fontSize` to the current tile dimensions so resizes look right too. Label is a compact centered tag (~75% of tile width, clamped 140–360px) rather than a banner spanning the whole tile
- Talisman tile label visual anchoring: Foundry paints the DrawingsLayer BEFORE the TilesLayer in the canvas group, so a label drawing positioned inside the tile's bounding box was hidden behind the tile texture entirely. Label is now placed immediately below the tile with an 8px gap — small enough to read as "attached to the talisman", large enough that any letterbox padding around the tile's visible texture doesn't visually clip into it
- Talisman tile label: removed the blue border (`strokeColor: '#00bfff'`) around the label box. Text routinely overflowed the shape's fixed width and the box clipped through the text awkwardly; without the border it just reads as a dark caption under the talisman
- Initiative formula: changed `CONFIG.Combat.initiative` from `'1d20 + @abilities.dex.mod'` (which references a path that doesn't exist on CAIN actors) to a plain `'1d6'`. CAIN doesn't use Foundry's initiative system by default, but tools like the Carousel Combat Tracker do — a 1d6 fits CAIN's d6-based dice pool, and the previous formula was likely throwing roll errors when those modules tried to use it
- Other Powers card-open bug: clicking a power under "Other Powers" (typically homebrew / unlinked blasphemy powers) opened the wrong card — always the first one in the list. The Passives and Powers sections wrap the name + action buttons in an `<h4>`, but the Other Powers section had the span and buttons directly inside `<li>`. The shared click handler does `event.target.parentElement.parentElement.querySelector('.power-description-card')`; in Other Powers that walked up past the `<li>` to the `<ul>` and picked up the FIRST power's card. Added the matching `<h4>` wrapper so the traversal lands on the same `<li>` as the clicked power
- Player edit-power button: added a pencil ("Edit this character's copy") icon next to the chat/remove buttons on every blasphemy power on the character sheet — for passives, active powers, and unlinked powers. Click opens the editable per-character item sheet. If the power is still legacy (pre-embed-on-add, never converted), clicking edit silently embeds a copy first so the edit lands on the player's own version and can't leak to other characters who have the same power

## 1.3.19

- Foundry v14 compatibility: namespaced `mergeObject`, `TextEditor`, and `duplicate` calls so they resolve under both v12 (global) and v13+ (`foundry.utils` / `foundry.applications.ux.TextEditor.implementation`)
- Removed the deprecated `async: true` option from `Roll#evaluate` and `TextEditor.enrichHTML` calls
- Awaited `actor.update` calls in `_performRoll` to prevent overlapping data-preparation cycles that triggered "ActiveEffect application phase already completed" errors
- Player Overview: moved tab-switching out of an inline `<script>` in the template into the V1 `activateListeners` lifecycle. Fixes the `Cannot read properties of null (reading 'classList')` error when no players have assigned actors, and stops click handlers stacking on every re-render
- Actor creation: removed the no-op `prepareData`/`prepareBaseData`/`prepareDerivedData` overrides on `CainActor` and the no-op `prepareData` on `CainItem`. Their empty bodies skipped `super`, leaving the v13+ ActiveEffect phase tracker uninitialized and producing `Cannot set properties of undefined (setting 'initial')` on `new` actor construction
- Actor creation: consolidated per-type default image and prototype-token defaults into a single `updateSource` call inside `CainActor#_preCreate`; removed the corresponding `_preCreate` overrides from `CainCharacter` and `CainNPC` data models. Multiple `updateSource` calls during `_preCreate` retriggered the data-prep cycle in v13+ and caused cascading "ActiveEffect application phase already completed" errors
- Domain item: `CainDomain#_preCreate` no longer unconditionally overwrites the source `img` with the ogre default. It now only sets the default when no image was explicitly provided. Previously, importing the Domains compendium caused every domain to display the ogre icon regardless of its source image
- Scene-control tooltips: in v14 the core scene-control handler silently drops clicks while `canvas.ready` is `false` (no active scene). The Divine Agony / Hunt Tracker / Dice Roller tool titles now append " — activate a scene to enable" when there's no scene, so hovering explains the non-response instead of looking broken
- Dice Roller scene-control: switched the v13/v14 branch from the deprecated `onClick` to `onChange` (the `onClick` deprecation warning is scheduled to remove the option in v15); the v12 branch keeps `onClick`
- CAT placeholder parser: the `{<CAT> type modifier}` regex required a modifier, so power text like Ardence's Sabre's `{<CAT> distance_normal}` (no modifier) fell through and the literal placeholder was shown. The modifier capture is now optional and defaults to `0`, matching the apparent author intent — Sabre now renders the CAT value the same way Fury does
- Psyche Burst dialog: added "Blast" to the Common Psyche Uses dropdown
- Multi-actor sheet sync bug: skill / psyche-burst / xp / advancement checkbox `id` attributes in the character sheet templates were not unique across sheets. When a single player had two character sheets open at once, the browser's `<label for="...">` lookup matched the first element in the document, so clicking a checkbox label on sheet 2 fired the change event on sheet 1's checkbox and wrote the update to actor 1. All four affected `id`/`for` pairs are now suffixed with `{{@root.actor.id}}` to make them unique per sheet. (Used `@root` rather than a relative path because the `range` and `each` helpers swap the Handlebars context, which would make a relative `{{actor.id}}` resolve to undefined.) This also explains the previously reported "Psyche Burst toggle sometimes does nothing or resets to full" symptom when multiple sheets were open
